# Acceptance Criteria — llm-first-reporting

## Must-Pass Criteria

| # | Criterion | How to Verify |
|---|-----------|---------------|
| AC-1 | Produces a single report file in the project directory | Output file exists with correct name for report type |
| AC-2 | Report is written in natural prose, not data tables | Read the output — it should read like a document, not a spreadsheet |
| AC-3 | Bracket tags are sparse — only on actionable/traceable items | Count bracket tags; should be significantly fewer than total entities in project |
| AC-4 | No inline markdown links in report body | Grep for `](` patterns — should find none in body sections |
| AC-5 | Report is tailored to stated audience | Language, detail level, and emphasis match the audience description |
| AC-6 | Bracket tags resolve to valid IDs in the structured project files | Every `[ACT-*]`, `[DEC-*]`, etc. in the report maps to an entry in the corresponding file |
| AC-7 | Skill is read-only — no project files modified (except the report) | Compare file checksums before/after report generation |
| AC-8 | Loads `references/formatting-conventions.md` during execution | Output follows all formatting convention rules |
| AC-9 | Phase 0 access gate blocks execution if methodology repo inaccessible | When gim-home/llm-first-specs is unreachable, skill stops with auth instructions |

## Test Scenarios

### Scenario 1: Status brief from an init scaffold
- **Input:** A project directory with 9 scaffold files, report type "status brief", audience "VP of Engineering"
- **Expected:** STATUS_BRIEF.md produced; prose-first format; bracket tags only on key blocked items and decisions; no more than 1 page of content
- **Validates:** AC-1, AC-2, AC-3, AC-4, AC-5

### Scenario 1b: V-Team brief from an init scaffold
- **Input:** A project directory with scaffold files, report type "status brief", focus "V-Team brief", audience "v-team"
- **Expected:** STATUS_BRIEF.md produced; three sections (Status, Action Items, Risks); half-page or less; bulleted items with owners; no background context; risks tagged with severity
- **Validates:** AC-1, AC-2, AC-3, AC-5

### Scenario 2: Audit summary from audit output
- **Input:** A project directory containing AUDIT_REPORT.md from `/llm-first-audit`, report type "audit summary", audience "project team"
- **Expected:** AUDIT_SUMMARY.md produced; narrative explaining scores and findings; bracket tags on specific improvement items; actionable next steps
- **Validates:** AC-1, AC-2, AC-5, AC-8

### Scenario 3: Bracket tag traceability
- **Input:** Any valid project directory, any report type
- **Expected:** Every bracket tag `[ACT-*]`, `[DEC-*]`, etc. in the report corresponds to a valid entry in the structured files
- **Validates:** AC-6, AC-3

### Scenario 4: Read-only guarantee
- **Input:** Any project directory
- **Expected:** Only the report file is created; all other files unchanged
- **Validates:** AC-7

### Scenario 5: Round-trip — annotated report fed back
- **Input:** A report previously generated by this skill, with human annotations next to bracket tags
- **Expected:** An agent can identify the bracket tags, resolve them to structured file entries, and process the annotations
- **Validates:** AC-6 (indirectly — verifies tags are well-formed for round-trip)

## Additional Criteria

| # | Criterion | How to Verify |
|---|-----------|---------------|
| AC-10 | V-Team briefs have exactly 3 sections (Status, Action Items, Risks) and are ≤ half a page | Count H2 headers — must be exactly 3 with the specified names; total content ≤ ~250 words |
| AC-11 | All Critical/High consistency items are either mentioned or listed as out-of-scope | Cross-check every CON-\*/AMB-\*/GAP-\* with severity Critical or High against report content |
| AC-12 | Data quality limitations from source files are disclosed in the report | If any GAP-\* or source entry flags incomplete/unretrievable data, report must contain a disclosure section |

### Scenario 6: V-Team brief format enforcement
- **Input:** A project directory with scaffold files, report type "status brief", focus "V-Team brief"
- **Expected:** Report contains exactly 3 H2 sections: Status, Action Items, Risks. No Summary, no Decisions Pending, no Next Check-In. Total length ≤ half a page (~250 words). Risks include severity tags.
- **Validates:** AC-10, AC-5

### Scenario 7: Critical item reconciliation
- **Input:** A project with "At Risk" activities and Critical/High consistency items
- **Expected:** Every at-risk or critical-not-started activity appears in the report. Every Critical/High CON-\*/AMB-\*/GAP-\* is either mentioned or explicitly noted as out-of-scope with justification.
- **Validates:** AC-11

### Scenario 8: Data quality disclosure
- **Input:** A project where SOURCES.md or CONSISTENCY_REPORT.md flags unretrievable or incomplete data
- **Expected:** Report includes a data limitations disclosure at the end
- **Validates:** AC-12
