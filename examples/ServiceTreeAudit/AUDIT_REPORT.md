## 2. FAILURE MODE VULNERABILITY ASSESSMENT

### FM-1: Missing or Unstable Identifiers
**Rating: PARTIALLY PROTECTED**

The scaffold assigns stable IDs to definitions, entities, decisions, sources, discoveries, conflicts, ambiguities, and gaps — an impressive 8 namespaces. However, the **most operationally important items lack IDs**: the 22+ tracker rows (remediation activities). These are referenced throughout as "Row 7", "Tracker Row 17", "Rows 14-15". If the Excel tracker is reordered, every reference in the scaffold breaks. The whitepaper (Section "Failure Modes", item 1, line 512-520) is explicit: *"Introduce stable IDs; treat them as permanent primary keys."*

**Specific gaps:**
- Tracker rows: "Row 7" should be `ACT-007` or `ACT-LARGE-SERVICES`
- Evidence items: No evidence registry with `EVD-*` IDs
- "Tim Buzard" (CONSISTENCY_REPORT.md line 121): mentioned but no ENT-* entry
- "Rajesh Jha", "Parul" (RELATIONSHIPS.md line 48): mentioned but no ENT-* entries

---

### FM-2: Implicit Relationships
**Rating: PARTIALLY PROTECTED**

RELATIONSHIPS.md is a dedicated relationship file — which is excellent and goes beyond what most whitepaper examples demonstrate. However, the primary relationship structure (Finding→Commitment→Activity chain, lines 11-39) is an **ASCII art tree inside a code block**, not navigable markdown links. An LLM can read it, but cannot click-traverse it. The whitepaper (line 530-533) prescribes: *"Use explicit links: `Depends On: [WS-2.1](#ws-2-1)`. Treat links as 'foreign keys' in a relational graph."*

Additionally, the Ownership Map and Decision Impact Map in RELATIONSHIPS.md use same-file anchors (`#ent-daysha`) for entities defined in ENTITIES.md — these are broken foreign keys.

---

### FM-3: Overloaded Definitions
**Rating: PROTECTED** ✅

This is where the scaffold excels. Three critical terms (`DEF-CRITICAL-FIELD-001`, `DEF-GOVERNANCE-001`, `DEF-SERVICE-OWNER-001`) are explicitly flagged as **ambiguous** with inline ⚠️ warnings pointing to specific AMB-* entries in CONSISTENCY_REPORT.md. Rather than hiding the overload, the scaffold surfaces it as a first-class problem to resolve. This exceeds the whitepaper's "one concept = one definition" guidance by adding an intermediate state: "this definition is overloaded AND we know it AND here's the resolution path."

### FM-5: Mixing Authoritative Truth with Draft Brainstorming
**Rating: PARTIALLY PROTECTED**

The scaffold separates concerns well across files (canonical definitions in GLOSSARY.md, conflicts in CONSISTENCY_REPORT.md). However:
- There is no `drafts/` or `notes/` directory for working content
- The CONSISTENCY_REPORT.md mixes "things we know" with "things we need to figure out" — which is appropriate for that file's purpose, but someone editing could inadvertently add speculative content to GLOSSARY.md or ENTITIES.md
- No file-level markers indicating "this file is canonical" vs. "this file contains unresolved items"

### FM-6: Unstructured Word or Loop Documents in the Authoring Path
**Rating: VULNERABLE** ⚠️

This is the scaffold's biggest process risk. The canonical sources are:
- `SRC-AUDIT-REPORT`: Word document
- `SRC-MAP`: Word document (with 3 copies per CON-1)
- `SRC-KEY-ACTIVITIES-TRACKER`: Excel workbook
- `SRC-KEY-ACTIVITIES-NARRATIVE`: Word document

The scaffold extracts knowledge from these into markdown — which is correct. But there is **no documented process for what happens when the Word/Excel sources are updated**. Will someone update the tracker Excel and forget to update ENTITIES.md? The whitepaper (lines 573-581) prescribes: *"Markdown + Git = canonical. Provide Word/PDF as satellite review artifacts only."*

The scaffold needs to declare: **this markdown is now the POR. The Word/Excel files are historical inputs, not living sources.**

## 4. IMPROVEMENT OPPORTUNITIES

### 4.1 — Add Stable IDs to Tracker Activities
- **What:** Create `ACT-*` IDs for each tracker row. E.g., `ACT-001: Document consumption programs`, `ACT-007: Break up large/monolithic services`, `ACT-017: Upgrade metadata policy engine`.
- **Why:** FM-1 (Missing Identifiers). "Row 7" is fragile — if the tracker is reordered, all references break. The whitepaper treats IDs as "permanent primary keys" (line 519).
- **Impact:** **HIGH** — Activities are the operational core. Every status update, every audit conversation, every evidence link references them.
- **Example:**
  ```markdown
  **ACT-007** {#act-007}: Break up large/monolithic services
  - **MAP Section:** [DEC-MAP-SECTION-2](DECISIONS.md#dec-map-section-2)
  - **Owner:** [ENT-ANDREW](ENTITIES.md#ent-andrew) (per [DEC-ROW7-OWNER](DECISIONS.md#dec-row7-owner))
  - **Status:** In Progress
  - **Evidence:** (none linked)
  - **Original ETA:** FY25 H2
  - **Risk:** Flagged as unlikely to meet deadline
  ```

### 4.2 — Fix All Cross-File Links to Use Full Paths
- **What:** Replace all bare-anchor cross-file references (`[ENT-DAYSHA](#ent-daysha)`) with full-path references (`[ENT-DAYSHA](ENTITIES.md#ent-daysha)`).
- **Why:** FM-2 (Implicit Relationships). The whitepaper treats links as "foreign keys" (line 533). A foreign key that doesn't resolve is a broken constraint.
- **Impact:** **HIGH** — ~30+ links across RELATIONSHIPS.md, DECISIONS.md, and GLOSSARY.md are affected. Fixes navigability for both humans and LLMs.
- **Example (RELATIONSHIPS.md Ownership Map):**

## 5. WHAT THE SCAFFOLD DOES WELL

### 5.1 Ambiguity Surfacing (Novel Pattern) ⭐
The scaffold's approach to overloaded definitions is **superior to the whitepaper's guidance**. Rather than simply saying "one concept = one definition," the scaffold introduces a three-layer pattern:
1. Define the term in GLOSSARY.md with a stable ID
2. Flag it with ⚠️ inline when ambiguous
3. Point to a specific AMB-* entry in CONSISTENCY_REPORT.md with multiple competing definitions, resolution path, and owner

This is a **novel contribution** not demonstrated in any of the 5 case studies. It handles the real-world case where you can't resolve ambiguity immediately but need to prevent it from causing silent errors.

### 5.2 Discovery Provenance (Audit Trail)
DISCOVERY.md is an exceptional addition not present in the whitepaper's patterns. It answers "did anyone look for X?" by documenting:
- What was searched (with exact queries)
- What was found
- What was included/excluded (with rationale)
- What wasn't searched yet (with conditions for when to search)

This is critical for audit work and would be valuable in any case study domain.

### 5.3 Multi-File Separation of Concerns
The 8-file structure (PROJECT, GLOSSARY, ENTITIES, RELATIONSHIPS, DECISIONS, SOURCES, DISCOVERY, CONSISTENCY_REPORT) is a clean separation that enables focused updates. You can update ownership in ENTITIES.md without touching GLOSSARY.md. This is better than monolithic POR files (CS2's `por.md`).

### 5.4 Source Attribution
Every definition, entity, and decision includes source attribution (which meeting, which document). This creates traceability from structured knowledge back to raw source material — essential for audit work where provenance matters.

### 5.5 Status Taxonomy
GLOSSARY.md defines 6 status values (Not Started, In Progress, Complete, Blocked, Disputed, Replaced) with semantic precision. The "Disputed" and "Replaced" statuses are novel — they handle real-world situations (Kaushik's disputes, alternative approaches) that simple Done/Not Done taxonomies miss.

### 5.6 Consistency Report as First-Class Artifact
Elevating conflicts, ambiguities, and gaps to a dedicated file with stable IDs (CON-*, AMB-*, GAP-*), severity ratings, and resolution ownership is a strong pattern. It makes "what we don't know" as visible as "what we do know."

## 6. RECOMMENDED CHANGES TO THE SKILL DESIGN

Based on this audit, the `/llm-first-init` skill should be modified as follows:

### 6.1 — Generate Activity IDs, Not Row References
When extracting work items from source documents (trackers, MAPs), assign `ACT-*` stable IDs rather than referencing row numbers. Row references are the single most fragile element in the scaffold.

### 6.2 — Enforce Full-Path Cross-File Links
The skill should generate all cross-file references using the full-path format (`[ID](FILE.md#anchor)`) and never bare anchors (`[ID](#anchor)`) for IDs defined in other files. This should be validated as a post-generation check.

### 6.3 — Use Navigable Markdown for Relationship Trees
Replace ASCII art code-block trees with structured markdown (nested headers + bullets + links). Code blocks defeat the "explicit relationships as foreign keys" principle.

### 6.4 — Generate at Least One Projection
The skill should generate at least one audience-specific projection (e.g., `STATUS_BRIEF.md`) from the structured data. This demonstrates the projection pattern and gives the user a working example to extend.

### 6.5 — Add a Canonical Source Declaration
The skill should include a section in PROJECT.md declaring the markdown as the POR and specifying that original Word/Excel sources are archived inputs, not living documents.

### 6.6 — Add Anchors to All ID Occurrences
Every ID that appears in a definition position (not just a reference) should have a `{#id}` anchor. The DISCOVERY.md triage entries are missing anchors despite having IDs.

### 6.7 — Include Exit Criteria Template
Following CS4, the skill should generate an exit criteria section (even if initially empty) to prompt the user to define what "done" means.

### 6.8 — Include a Verification Prompt
Add a section to PROJECT.md or a standalone `VERIFY.md` with LLM prompts for re-running consistency checks:
```
Suggested verification prompts:
1. "Check all cross-file links — do all [ID](FILE.md#anchor) targets exist?"
2. "Find entities mentioned in prose but without ENT-* entries."
3. "List all ACT-* items without evidence links."
4. "Are there any GLOSSARY terms used in other files that don't match their canonical definition?"
```

### 6.9 — Document the Scaffold Schema
Include a brief schema reference (either in PROJECT.md or a dedicated file) that specifies the required and optional fields for each entity type. This helps both humans and LLMs add new entries consistently.

### 6.10 — Consider a Timeline/Milestone Projection
For projects with deadlines (like audit remediation), auto-generate a timeline view aggregating all dates from DECISIONS.md and activity entries into a single chronological view.

## Summary

The ServiceTreeAudit scaffold is a **strong implementation** of LLM-First principles — scoring 3.7/5 overall with exemplary consistency (5/5), definitions (5/5), and a novel ambiguity-surfacing pattern that exceeds the whitepaper. Its weaknesses are systematic and fixable: missing activity-level IDs, inconsistent cross-file links, code-block relationship trees, and no audience projections. The 10 improvements above are ranked by impact and would bring the scaffold to 4.5+/5 alignment.

**Top 3 actions (highest impact, lowest effort):**
1. Add `ACT-*` IDs to tracker activities (fixes FM-1, unblocks FM-10)
2. Fix cross-file links to use full paths (fixes FM-2, ~30 links)
3. Generate one audience projection (fixes FM-7, proves the pattern)
